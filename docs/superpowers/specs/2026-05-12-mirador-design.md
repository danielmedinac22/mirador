## 1. Vision

> **Mirador is the shared place where HTML generated by AI becomes a first-class, collaborative artifact.**

Markdown is the lingua franca between humans and AI agents today, but it strips out the visual richness that the agent can actually produce. AI can write beautiful HTML — presentations, dashboards, reports, mini-apps — but there is no native channel to *share, view, comment on, and re-edit* that output with other humans. People paste HTML into emails, screenshot it, or commit it to a repo nobody opens.

Mirador turns "the HTML the AI just produced" into a **link with a runtime** — viewable, commentable, forkable, AI-editable, and (eventually) multi-player.

---

## 2. Problem Statement

When Claude Code / Codex / Cursor generates an HTML artifact (dashboard, slide deck, report, prototype), the user faces friction to share it:

1. The recipient needs to clone, install dependencies, or open a static file.
2. There is no inline commenting, no version history, no "edit this with AI" path.
3. There is no security boundary — HTML often has scripts that the recipient does not want to run unsandboxed.
4. There is no concept of *teamwork* on top of the artifact: comments, forks, password gates, expiry, embed.

Existing tools (Notion, Obsidian, Reveal.js, TinyMCE forks, Vercel previews) each cover a slice but no product treats *AI-generated HTML* as the primary unit.

---

## 3. Solution Summary

A two-phase product:

**Phase A — MVP (this spec):** A CLI + web viewer that turns any HTML file into a shareable link in one command. Sandbox rendering, comments, fork, "edit with AI", versions, password, expiry. Designed to be triggered from a Claude Code skill or Codex prompt.

**Phase B (out of scope for this spec, noted only for direction):** Real-time multi-player editing (Yjs/CRDT), block-level visual editor, custom domains, team workspaces.

This spec covers Phase A only. Phase B will have its own spec.

---

## 4. MVP Scope

### In scope

1. **CLI tool** `mirador` (`mirador share <file>`, `mirador list`, `mirador delete <id>`)
2. **Claude Code skill** wrapping the CLI so the agent can call it after generating HTML
3. **Web app** at `mirador.app/d/<id>` (codename — final domain TBD by user) that renders the HTML in a sandboxed iframe
4. **API** for creating documents, fetching them, posting comments, creating new versions, forking
5. **Sandbox rendering** with iframe `srcdoc` + restrictive CSP
6. **Inline comments** anchored to a CSS selector or rectangular region
7. **"Edit with AI"** flow: prompt → backend calls Claude API → produces a new version (immutable)
8. **Fork**: copy a document into a new ID
9. **Versioning**: every save = new immutable snapshot, URL may point to `latest` or a version hash
10. **Sharing controls**: public / unlisted (link-only) / password-protected; optional expiry
11. **Self-hosted deployment** via `docker compose up` with SQLite + local disk storage

### Out of scope (Phase B)

- Real-time multiplayer (CRDT, presence cursors, shared selection)
- WYSIWYG block editor (Notion-style)
- Team workspaces and SSO
- Custom domains
- Analytics
- Embed widget
- Mobile apps
- Granular per-element permissions

### Explicitly not building

- A general-purpose CMS
- A general-purpose static site host (we are *not* Vercel)
- A Markdown editor (we are the layer *after* Markdown)
- A code IDE

### Tech stack (MVP)

- **Language:** TypeScript everywhere.
- **CLI:** Node 20+, `commander` for parsing, `clipboardy` for copy-to-clipboard.
- **Web:** Next.js 15 (App Router), Tailwind, shadcn/ui, Monaco editor for raw HTML editing.
- **API:** Hono on Node (self-host) and on Cloudflare Workers (cloud) — Hono runs on both, so one codebase.
- **DB:** SQLite (self-host) / Postgres via Neon (cloud). Drizzle ORM so the same code runs on both.
- **Storage:** local disk (self-host) / Cloudflare R2 (cloud), behind a small `Storage` interface.
- **AI:** Anthropic SDK for the "edit with AI" endpoint.
- **Tooling:** pnpm workspaces, Biome for lint+format, Vitest for tests, GitHub Actions for CI.

## 6. Data Model

### Tables

```
users (optional for MVP — link-token auth is primary)
  id              text PK
  email           text UNIQUE
  created_at      timestamptz

documents
  id              text PK              -- short id, e.g. "abc123de"
  owner_user_id   text NULL FK
  owner_email     text NULL            -- captured at create time if no account
  visibility      text                 -- 'public' | 'unlisted' | 'password'
  password_hash   text NULL            -- argon2; only when visibility='password'
  expires_at      timestamptz NULL
  parent_id       text NULL FK         -- if forked, points to source doc
  created_at      timestamptz
  deleted_at      timestamptz NULL

versions
  id              text PK              -- short hash
  document_id     text FK
  blob_key        text                 -- key in object storage
  size_bytes      int
  created_by      text NULL            -- user id or 'ai' or 'cli'
  edit_prompt     text NULL            -- if created via "edit with AI"
  created_at      timestamptz

document_latest_version
  document_id     text PK FK
  version_id      text FK

comments
  id              text PK
  document_id     text FK
  version_id      text FK              -- comments are pinned to a version
  anchor_selector text                 -- CSS path or 'doc' for global
  anchor_rect     jsonb NULL           -- {x,y,w,h} fallback
  author_email    text NULL
  body            text
  resolved        boolean
  created_at      timestamptz

edit_tokens
  token_hash      text PK
  document_id     text FK
  scope           text                 -- 'edit' | 'view'
  expires_at      timestamptz NULL
  revoked         boolean
  created_at      timestamptz
```

Notes:

- IDs are short, URL-safe (8–10 chars from `nanoid`).
- Versions are immutable; "save" = insert a new row.
- `document_latest_version` is the mutable pointer; updating it is the only mutation a writer can perform.
- The MVP supports anonymous-by-email creation (no signup needed). Accounts come later for "my documents" dashboards.

## 7. API Surface (MVP)

All JSON. Auth via `Authorization: Bearer <edit_token>` when needed; viewers can pass `?p=<password>` query for password-protected docs.

| Method | Path | Purpose |
|---|---|---|
| POST | `/v1/docs` | Create a document. Body: `{ html, visibility, password?, expires_at?, owner_email? }`. Returns `{ id, url, edit_token, version_id }`. |
| GET | `/v1/docs/:id` | Fetch metadata + latest version blob URL (signed). Honors visibility + password. |
| GET | `/v1/docs/:id/versions/:vid` | Fetch a specific version blob. |
| POST | `/v1/docs/:id/versions` | Save a new version. Body: `{ html, edit_prompt? }`. Requires edit token. |
| POST | `/v1/docs/:id/fork` | Fork. Returns `{ id, url, edit_token }`. Anyone with view access can fork unless `forkable: false`. |
| DELETE | `/v1/docs/:id` | Soft-delete. Requires edit token. |
| GET | `/v1/docs/:id/comments` | List comments. |
| POST | `/v1/docs/:id/comments` | Create comment. Body: `{ version_id, anchor_selector, anchor_rect?, body, author_email? }`. |
| PATCH | `/v1/comments/:cid` | Resolve/unresolve. |
| POST | `/v1/docs/:id/ai-edit` | "Edit with AI". Body: `{ prompt, base_version_id }`. Server calls Anthropic API, returns new version. |
| GET | `/v1/health` | Liveness. |

Rate limits (defaults; configurable):

- 60 requests/min per IP across all endpoints.
- 10 doc creations/min per IP.
- 5 AI edits/min per doc.
- Max HTML size: 5 MB per version.

## 8. Security Model

This is the highest-risk surface — arbitrary user-supplied HTML rendered in a browser.

### Rendering rules

1. **Iframe with `sandbox="allow-scripts"`** — no `allow-same-origin`. Scripts run in a null origin, so they cannot read cookies, call same-origin XHR, or touch the parent page.
2. **`srcdoc`** is preferred over `src` so the iframe content is delivered inline; this avoids leaking the storage URL to the script.
3. **CSP injected via a `<meta http-equiv="Content-Security-Policy">` tag prepended to the HTML** before rendering. Default policy:
   ```
   default-src 'none';
   script-src 'unsafe-inline' 'unsafe-eval';
   style-src 'unsafe-inline' https: data:;
   img-src https: data:;
   font-src https: data:;
   media-src https: data:;
   connect-src https:;
   frame-src https:;
   ```
   Rationale: scripts must be allowed inline (that's the whole point), but the sandbox null origin already prevents most cookie/storage abuse. The CSP blocks loading scripts from third-party URLs (`script-src` has no `https:`) to reduce supply-chain risk.
4. **Strict mode (opt-in)** uses DOMPurify on the server to strip `<script>`, `on*` handlers, and dangerous URL schemes. Default mode does *not* sanitize — it preserves AI output verbatim and relies on the sandbox.
5. **Top-frame navigation block:** the rendered HTML cannot escape the iframe — `allow-top-navigation` is not granted.
6. **No referer leakage:** the viewer sets `Referrer-Policy: no-referrer` on the iframe.

### Storage and access

- Doc blobs are served via signed URLs (short TTL, ~5 min) when not embedded as `srcdoc`. MVP uses `srcdoc` always; signed URLs reserved for the editor's "open in new tab" affordance.
- Password-protected docs: password is hashed with argon2; verification is server-side, never client-side.
- Edit tokens are 32-byte URL-safe random, hashed at rest, scoped to a single document.

### Abuse and rate

- Per-IP and per-doc rate limits as listed above.
- Optional content scan hook (interface only; no scanner shipped in MVP) so cloud deployment can plug in VirusTotal / a custom classifier later.

### Threats explicitly accepted in MVP

- A motivated attacker can still ship a phishing page inside the iframe. The path forward (v1.x) is reputation/warning banners on first view; not in MVP.
- Crypto miners running inside the sandbox — possible. Mitigated only by rate limits and abuse reports.

## 9. Claude Code Skill Behavior

The skill is what makes Mirador "AI-native". Manifest sketch:

```
~/.claude/skills/mirador/
├── SKILL.md           # description so Claude knows when to invoke
└── scripts/
    └── share.sh       # thin wrapper that calls `mirador share`
```

`SKILL.md` description (draft):

> Use this skill when you have just produced an HTML artifact (presentation, dashboard, report, mini-app) and the user wants to view or share it. Wraps the `mirador` CLI to upload the HTML and return a shareable URL with optional password and expiry.

Invocation contract from the agent's perspective:

- Input: path to a local `.html` file produced in the current session, plus optional `visibility` / `password` / `expires` flags.
- Output: a single line `https://mirador.app/d/<id>` (or self-hosted equivalent) plus the URL on the user's clipboard.

The skill does *not* try to be clever about parsing intent — it is a thin tool. Detection of "the user wants to share" is the calling agent's job.

## 10. Roadmap

### Phase A — MVP (this spec)

Milestones, in order:

1. **Skeleton end-to-end (local only)**
   - Monorepo scaffold, CLI stub, Next.js app, Hono API.
   - "Hello world" path: CLI uploads a hardcoded HTML, API stores in memory, web renders.
2. **Persistence**
   - Drizzle + SQLite, local disk storage for blobs.
3. **Real sandbox + CSP**
   - Iframe + srcdoc + CSP. Render arbitrary HTML safely.
4. **Versions + fork**
5. **Comments (global first, then anchored)**
6. **AI edit endpoint** wired to Anthropic SDK.
7. **Password + expiry + visibility**
8. **Claude Code skill packaged + tested end-to-end.**
9. **`docker-compose.yml` self-host story + README.**
10. **Hosted preview at `mirador.app` (cloud deploy on Workers + R2 + Neon).**

### Phase B (separate spec later)

Multiplayer (Yjs), block editor, team workspaces, custom domains, analytics, embed widget.

## 11. Open Source & Productization Strategy

- **Repo:** public on GitHub, monorepo.
- **Licenses:**
  - `@mirador/cli`, `@mirador/skill`, `@mirador/sdk`, `@mirador/sanitizer`: **MIT** (we want broad adoption).

- **Hosted (Mirador Cloud):**
  - Free tier: public docs, limited storage, no custom expiry, Mirador branding.
  - Pro: private docs, longer expiry, custom passwords, removed branding, AI edits included.
  - Team: workspaces, audit log, SSO.
- **Branding:** name `Mirador`, vibe calm/scenic. Logo and domain TBD by user; spec uses `mirador.app` as placeholder.

## 12. Success Criteria

The MVP is considered successful when:

1. A user can install the Claude Code skill, ask Claude to "generate a one-page report on X and share it", and within ~10 seconds receive a viewable URL.
2. Opening that URL in a fresh browser renders the HTML safely, with no console errors and no ability for the rendered HTML to escape the sandbox (verified by automated test).
3. The recipient can leave a comment without an account.
4. The recipient can fork and request an AI edit ("make the chart smaller"), producing a new version.
5. `docker compose up` on a fresh machine yields a working Mirador in under 5 minutes.
6. The repo is public on GitHub with clear contribution docs.
